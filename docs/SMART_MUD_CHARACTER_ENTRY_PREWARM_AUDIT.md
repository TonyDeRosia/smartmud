# Smart MUD Character Entry Prewarm Audit

Primary reference reviewed: Adventurer's Lair `src/comm.c`, `src/interpreter.c`, `src/db.c`, `src/handler.c`, and `src/limits.c` from `TonyDeRosia/tbamud_adventurers_lair` on GitHub. This is a behavioral audit only; Smart MUD intentionally does not copy C structs, descriptor handling, socket loops, macros, or file formats.

| Behavior | Reference file/function | Player-visible result | Smart MUD equivalent | Current Smart MUD status | Changes in this phase | Intentional architecture difference |
|---|---|---|---|---|---|---|
| Account/menu state | `comm.c` descriptor connection states and login loop | Before play, the descriptor carries lightweight menu/input state rather than full gameplay work. | FastAPI/Web session plus `runtime_sessions`; character select lists summaries via `list_characters`. | Character select already uses summary rows. | Confirmed no full inventory/equipment/effect load is added to account login. | Web/API sessions replace telnet descriptors. |
| Character selection | `comm.c` login/copyover recovery and enter-game path | Selected character transitions into live play. | `MudRuntime.enter_world`. | Enter existed but mixed load, save, view, and registration. | Added entry correlation id and immutable `CharacterEntryContext`. | Python service coordinator instead of C nanny state. |
| Character load | `db.c` player load routines and `comm.c` copyover recovery | Character data is loaded once when play begins/recovery occurs. | `_resident_character` and `enter_world`. | Resident map existed. | Entry loads persisted character once, registers resident object, and avoids entry-time character save. | SQLite recovery store replaces pfile. |
| Descriptor/live-character ownership | `comm.c` `descriptor_list`, `d->character`, `enter_player_game` | Commands operate on the descriptor-owned live character. | `active_characters`, `session_active_character`, `character_session_ids`, `ActorRegistry`. | Active map existed but room presence scans could reload SQLite. | Room delivery now iterates resident active characters only. | Session ids and runtime actors replace C descriptor pointers. |
| Command processing | `interpreter.c` command interpreter | Input dispatch uses in-memory character state. | `MudCommandEngine` and `MudRuntime.mud_input`. | Read-only save suppression existed. | Read-only displays consult the projection registry before synchronous rebuild. | Deterministic Python command registry replaces C command table. |
| Prompt generation | prompt-related code used from `comm.c` output path | Prompt reflects live HP/mana/move/state. | `prompt_snapshot`, `render_prompt`. | Prompt rebuilt repeatedly. | Prompt projection is cached and targeted-invalidated by resource/location changes. | Semantic HTML/plain prompt snapshot replaces telnet-only string. |
| Output queues | `comm.c` input/output queues and `process_output` | Player sees direct command output and queued room/async messages. | Direct command response plus async room output queues. | Already split direct vs async output. | Warm-up tasks never enqueue player output. | HTTP polling/event endpoints replace socket writes. |
| Heartbeat | `comm.c` game loop pulses and heartbeat functions | Timed systems progress independently of player display commands. | `runtime_pulse`, living world, combat runtime. | Pulse existed. | Added server-owned autosave scan helper for active dirty characters. | FastAPI/service pulse replaces one blocking C socket loop. |
| Timed systems | `limits.c`, `handler.c`, combat/effect update hooks | Regen, effects, and cleanup occur from server time. | `RuntimeResourceService`, `AbilityExecutionService`, combat/effects services. | Runtime services existed. | Entry context records effect/cooldown snapshots and projection source versions. | Services own domain data instead of monolithic char macros. |
| Autosave | `comm.c` shutdown save and periodic heartbeat save patterns | Persistence protects recovery without saving every display. | `ActiveCharacterAutosaveService`, `save_character_if_dirty`. | Dirty save coordinator existed. | Autosave scans only dirty active characters, coalesces reasons, tracks success/failure metrics. | Explicit service object replaces C heartbeat save calls. |
| Disconnect/cleanup | `comm.c` socket close and copyover cleanup | Quit/disconnect flushes and removes live ownership. | Quit command session transition cleanup. | Active character removal existed. | Quit cancels warmups and evicts private projection cache before unregistering actor. | Reconnect grace remains disabled unless configured later. |

## Reference summary

`comm.c` documents the central loop as responsible for accepting connections, polling input, sending queued output, and running heartbeat functions. Copyover recovery loads the character, attaches it to a descriptor, enters play, and renders the room. The reviewed behavior supports Smart MUD's equivalent model: keep account/menu state light, load the selected character at entry, keep it resident while commands run, generate prompts from live state, and use persistence for recovery/save boundaries rather than for every read-only command.
