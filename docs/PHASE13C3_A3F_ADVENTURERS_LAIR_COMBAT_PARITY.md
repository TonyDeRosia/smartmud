# Phase 13C3-A3G Adventurer's Lair Combat Parity Audit

Primary reference reviewed: Adventurer's Lair/tbaMUD combat, spell, handler, class, informative, offensive, constants, structs, and utils sources, especially `fight.c` and `spells.c`. Smart MUD implements equivalent player-facing behavior with SQLite-backed runtime services rather than copying C formulas, globals, structs, arrays, or file formats.

## Audited combat and damage sources

| Source | Routing after A3G | Notes |
|---|---|---|
| Player melee/unarmed/ranged attack | `start_player_attack` / `_execute_attack` -> `submit_action` | Command handlers do not calculate combat numbers. |
| Queued automatic round attack | `process_encounter_round` -> `_execute_attack` -> `submit_action` | Round scheduler only chooses order/readiness. |
| NPC/mob attack | `start_actor_attack` or round processing -> `submit_action` | Entity stats/equipment/affects feed `CombatStatService`. |
| Controller attack/defend/flee/wait | Controller-facing methods queue safe action types only; forbidden result fields are rejected in `submit_action`. | Denial publishes `combat_action_validation_denied`. |
| Ability direct damage/spell attack | `AbilityExecutionService._apply_damage_component` builds `CombatActionRequest(source_type='ability')` when runtime combat is available. | Ability service no longer owns final damage for runtime gameplay. |
| Ability healing | `AbilityExecutionService.apply_healing` -> `RuntimeResourceService`; `CombatActionRequest(attack_kind='healing')` is supported for combat-history healing. | Ordinary dead targets are not revived. |
| DOT/effect damage | `CombatActionRequest(source_type='effect', attack_kind='damage_over_time')` -> `submit_action`. | Source/effect id is preserved for credit. |
| Environmental/survival/trap/hazard/scripted damage | `CombatActionRequest(source_type='environment'/'survival'/'trap'/'hazard'/'script', attack_kind='environmental_damage' or equivalent)` -> `submit_action`. | Environment deaths must not use a player attacker for reward credit. |
| Reactive/counter/thorns | Correlated `CombatActionRequest` with `parent_action_id`, `reaction_chain_id`, and `reaction_depth`. | Safe depth fields are present to prevent loops. |
| Admin damage | Should use a canonical action source_type `admin` so diagnostics and lifecycle are durable. | Admin mutation commands were not added. |

## Duplicate-action guarantee

`combat_action_requests` is the durable idempotency claim table. `submit_action` claims before execution, returns the stored completed result for duplicate completed action ids, and records the result JSON after resource mutation, combat history, lifecycle, corpse, reward, and respawn processing. Lifecycle side effects use stable transition ids and step status columns (`defeat`, `death`, `corpse`, `reward`, `respawn`, `combat_end`) so repeated lethal processing returns the same corpse/reward/respawn markers instead of duplicating them.

Manual Windows acceptance was **not performed** in this non-interactive Linux container.
